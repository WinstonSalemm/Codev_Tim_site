## Overview

Codev ERP — система планирования ресурсов предприятия (ERP) для операционной автоматизации бизнеса. Платформа объединяет учёт запасов, финансы и операционную отчётность в одной транзакционной системе на базе .NET MAUI, ASP.NET Core и PostgreSQL (Clean Architecture). Репозиторий: [WinstonSalemm/FinancialApp](https://github.com/WinstonSalemm/FinancialApp). Статус: In Development.

Система ориентирована на организации, которые переросли табличные процессы, но не требуют полноценного мультитenant ERP SaaS на старте. Область охвата ограничена ключевыми операционными доменами: контроль запасов, жизненный цикл заказа, приём поставок и управленческие панели. Отчётность читает те же транзакционные данные, пока нагрузка не оправдает выделенные read-модели.

Приоритет проектирования — целостность данных и аудируемость, а не широта функционала. Каждый путь записи проходит серверную валидацию и авторизацию. Клиент отображает состояние; бизнес-правила на клиенте не хранятся.

## Problem Statement

Операционные команды среднего сегмента обычно ведут запасы, заказы и закупки в разрозненных инструментах. Таблицы хранят остатки. Мессенджеры передают подтверждения заказов. Бухгалтерские выгрузки приходят спустя дни после события. Результат — постоянный разрыв сверки: ни одно число не принимается без ручной проверки.

Три типовых сбоя формируют требование к выделенной ERP-платформе.

**Фрагментированный источник истины.** Когда остатки в одном файле, а подтверждённые заказы в другом, перепродажи и фантомные остатки становятся нормой. Персонал тратит циклы на сверку вместо исполнения.

**Запаздывающая отчётность.** Управленческие решения опираются на устаревшие выгрузки. К моменту обновления панели условия уже изменились. Реактивное планирование вытесняет операционный контроль.

**Долг интеграций при масштабировании.** Каждый новый канал (маркетплace, склад, платёжный провайдер) добавляет point-to-point адаптер. Без канонической доменной модели адаптеры дублируют логику и расходятся друг с другом.

Codev ERP устраняет эти ограничения централизацией доменного состояния, принуждением инвариантов на границе API и операционными представлениями из того же ledger, что обслуживает запись.

## Business Context

Начальный контекст развёртывания — одна организация с одним или несколькими складами и ограниченным каталогом. Пользователи: складской персонал, операторы закупок, управленческие ревьюеры. Роли соответствуют операционным обязанностям: приём запасов, фиксация заказов, согласование дозакупок, чтение панелей.

Регуляторные и аудиторские требования распространяются на движение запасов и историю заказов. Система фиксирует, кто и когда изменил данные. Soft delete для ledger-сущностей не применяется; исправления добавляют audit-записи, а не стирают историю.

Операционный ритм пакетный в рабочие часы: утренняя сверка остатков, дневное исполнение заказов, вечерняя сверка. Пиковая конкурентность умеренная. Для путей записи важнее согласованность, чем минимальная задержка.

Географический охват начинается с локальной валюты и часового пояса. Интернационализация отложена до стабилизации доменных модулей. Внешние интеграции (бухгалтерия, перевозчики) подключаются через явные adapter-модули, а не ad hoc скрипты.

<Callout variant="info" title="Граница области">
  Codev ERP не является универсальной CRM, HR-системой или MES. Эти домены вне
  области до достижения production-стабильности модулей запасов и заказов.
</Callout>

## System Blueprint

Codev ERP построен как modular monolith: один deployable unit со строгими внутренними границами модулей. Решение обменивает независимое масштабирование сервисов на операционную простоту на ранней стадии. Межмодульные вызовы in-process; транзакции БД могут охватывать связанные домены, когда инварианты требуют атомарности.

Поток запросов вертикальный: Client → API → Services → Database → Infrastructure. Отдельный gateway-слой на текущем этапе отсутствует; TLS termination и маршрутизация — на edge инфраструктуры.

**Client layer.** .NET MAUI рендерит модульные desktop-представления. Управление состоянием опирается на server-fetched data с инвалидацией кэша при мутациях. Клиент не вычисляет доступность запасов без ответа сервера.

**API layer.** ASP.NET Core хостит версионированные REST endpoints. Аутентификация — bearer tokens с role claims. Авторизация выполняется до handler. Ошибки валидации возвращаются как structured problem details.

**Services layer.** Доменные модули инкапсулируют бизнес-правила. Inventory владеет остатками и резервированиями. Orders — переходами жизненного цикла. Procurement — заявками поставщикам и сопоставлением приёмки. Reporting читает через service interfaces, не напрямую из handlers.

**Database layer.** PostgreSQL хранит нормализованные транзакционные схемы. Миграции версионированы и применяются при deploy. Критические таблицы содержат `created_at`, `updated_at` и ссылки на actor для audit.

**Infrastructure layer.** Docker упаковывает API и БД для воспроизводимых сред. Production — single-region deployment с scheduled backups и мониторингом health endpoints.

## Technology Stack

| Layer   | Technology                    | Role                                                             |
| ------- | ----------------------------- | ---------------------------------------------------------------- |
| Client  | .NET MAUI, C#                 | Desktop module UI, role-aware views                              |
| API     | ASP.NET Core                  | REST-сервисы, middleware pipeline, dependency injection          |
| Data    | PostgreSQL                    | ACID-транзакции, реляционная целостность, индексированные отчёты |
| Runtime | Docker                        | Паритет dev/prod, локальный stack на compose                     |
| Tooling | TypeScript (client), C# (API) | Статическая типизация на границах                                |

**.NET MAUI** обеспечивает native desktop UI для плотных операционных экранов.

**ASP.NET Core** даёт зрелый middleware для auth, logging и validation. Фреймворк подходит для long-running server workloads и структурированной обработки исключений.

**PostgreSQL** выбран для транзакционных ERP-нагрузок: foreign keys, constraints, window functions для операционных отчётов. Document stores отклонены из-за join-heavy reporting.

**Docker** стандартизирует локальную разработку и артефакты развёртывания. API и БД работают как связанные контейнеры с явной конфигурацией портов и volumes.

Message broker в текущий stack не входит. Асинхронная работа — через in-process background jobs, пока outbox или queue не подтверждены нагрузкой в production.

## Roadmap

Разработка идёт в четыре фазы. Выход из фазы — только после прохождения acceptance criteria в staging.

**Phase 1 — Platform foundation (текущая).** Authentication, role model, pipeline миграций БД, health endpoints, CI build. Acceptance: authenticated API requests с audit logging; миграции применяются на пустой и существующей БД.

**Phase 2 — Core domains.** Модуль Inventory (остатки, резервы, warehouse scoping) и Orders (create, fulfill, cancel со state machine). Acceptance: end-to-end order flow атомарно уменьшает stock; недопустимые transitions отклоняются с явными ошибками.

**Phase 3 — Procurement and reporting.** Supplier intake, goods receipt против purchase orders, operational dashboards на live transactional data. Acceptance: KPI на dashboard совпадают с ручной SQL-проверкой в заданном tolerance.

**Phase 4 — Production readiness.** Проверка backup, performance baseline при ожидаемой concurrency, security review, deployment runbook. Acceptance: staged production cutover с документированной и проверенной процедурой rollback.

Version numbering начинается с v0.1.0 после Phase 2. Версионирование public API — semver на REST surface.

## Lessons Recorded

**Schema before features.** Ранние попытки ERP, начинавшиеся с UI, давали migration debt за недели. Сначала моделировались inventory и order entities; экраны следовали схеме, а не наоборот.

**State machines принадлежат services, не database.** Правила жизненного цикла заказа — в application code с явными transition tables. Database triggers отклонены: они скрывают поведение от code review и тестов.

**Reporting без read models допустим, пока load не доказан.** Начальные dashboards читают transactional tables по индексированным путям. Преждевременный CQRS задержал бы MVP без измеренной read/write contention.

**Audit columns обязательны.** Каждая ledger-affecting строка фиксирует actor и timestamp. Отладка production расхождений без audit data обходилась дороже storage overhead.

**Module boundaries требуют enforcement, не только документации.** Cross-module imports ограничены структурой проекта. Прямой repository access между доменами блокируется на code review.

**Operator-facing errors должны называть constraint.** Generic failure messages увеличивают support load. API responses включают machine-readable codes и human-readable описания ограничений (например, insufficient stock for reservation).
