## Overview

Poj Pro Operations Platform is the production internal business system for OOO «Poj Pro». A .NET MAUI desktop client connects to an ASP.NET Core API deployed on Railway, covering sales, warehouse, finance, contracts, and manager KPIs. The client auto-detects online vs offline mode and syncs queued operations from local SQLite when connectivity returns. Status: Production.

Repository: [WinstonSalemm/projectApp](https://github.com/WinstonSalemm/projectApp). Public catalog and corporate content live in [`poj-pro-site`](https://www.poj-pro.uz) ([Poj-Pro](https://github.com/WinstonSalemm/Poj-Pro)). Contract automation extends through `poj-pro-api-contracts` and `poj-pro-telegram-bots`.

## Problem Statement

Sales, warehouse, and finance workflows ran across spreadsheets, messaging apps, and ad hoc tools. Staff needed one authoritative system for cash and card sales, reservations, debt tracking, ND-40/IM-40 supply intake, tax analytics (UZ), and contract references — with reliable operation when internet or the Railway API is temporarily unavailable.

## Business Context

Users are internal operators, managers, and warehouse staff at OOO «Poj Pro». Usage peaks during business hours in Tashkent. The desktop client is the daily operations surface; the public website at https://www.poj-pro.uz serves B2B buyers separately.

<Callout variant="info" title="Scope boundary">
  This platform is not the public catalog website. Customer-facing web content
  and SEO are handled by `poj-pro-site`. Bots and contract HTTP services remain
  in their dedicated repositories.
</Callout>

## System Blueprint

Architecture follows a three-tier pattern: MAUI client, ASP.NET Core API on Railway, and persistent storage (PostgreSQL in production, SQLite offline on the client).

On launch the client probes API availability. Online mode writes immediately; offline mode queues operations and syncs when the server is reachable again.

## Technology Stack

| Layer      | Technology          | Role                                              |
| ---------- | ------------------- | ------------------------------------------------- |
| Client     | .NET MAUI, .NET 8   | Windows desktop UI, offline queue, sync indicator |
| API        | ASP.NET Core        | Business logic, REST boundary, Railway deploy     |
| Offline    | SQLite              | Local persistence and operation queue             |
| Production | Railway, PostgreSQL | Hosted API and primary datastore                  |

**.NET MAUI** delivers a native desktop experience for high-volume daily entry. **Offline-first sync** keeps sales and warehouse flows usable during connectivity gaps.

## Trade-offs

**Desktop MAUI vs web admin.** Desktop chosen for operator speed, peripheral integration, and offline resilience on the shop floor.

**Monolith API vs microservices.** Single ASP.NET Core deploy simplifies sync and transactional consistency for sales and stock.

**Separate public site repo.** Marketing catalog at poj-pro.uz deploys independently from internal operations (`poj-pro-site`).

## Roadmap

**Maintained.** Sales and warehouse modules, offline sync reliability, Railway API stability.

**Planned.** Deeper integration hooks with `poj-pro-api-contracts` for contract state in the client.

**Deferred.** Cross-platform MAUI targets beyond Windows desktop.

## Interface Record

Windows desktop application only. Primary areas: sales (cash, card, Click, reservation, debt), warehouse and supplies, clients and debtors, contracts, finance and cash desks, tax analytics, manager KPIs. Sync status shown in the top-right indicator (online, offline, pending queue, syncing).

## Lessons Recorded

**Offline queue must be visible.** Operators trust the system when pending sync count is always shown.

**Public web and internal ops are different products.** Mixing catalog SEO with ERP screens increased release risk; split into `Poj-Pro` and `projectApp`.

**API URL is environment config.** Railway base URL lives in client `appsettings.json` for controlled environment switches.
