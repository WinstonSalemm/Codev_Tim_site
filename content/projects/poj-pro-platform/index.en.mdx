## Overview

Poj Pro Corporate Platform is a production B2B web system for OOO «Poj Pro», a corporate supplier of fire safety, security, and lightning protection equipment. The platform publishes a large product catalog, multilingual corporate content, and administrative tooling using React, ASP.NET Core, and PostgreSQL. Status: Production at https://www.poj-pro.uz.

The system serves external B2B buyers and internal staff who manage catalog data, company news, and corporate information. It is not a marketing landing page. Catalog depth, category hierarchy, search, and localized content drive daily usage. Sales operations depend on accurate product data remaining available during business hours.

Production operation began under official employment (May 2025 – February 2026) and continues under ongoing cooperation. The platform handles real transaction-supporting workflows: product discovery, specification review, and content updates aligned with inventory and sales activity at approximately 1 billion UZS monthly company turnover.

## Problem Statement

OOO «Poj Pro» sells technical equipment through B2B channels. Buyers evaluate specifications, compare categories, and request quotes based on catalog data. Before the platform, product information was distributed across static pages, unstructured files, and manual updates. Sales staff duplicated answers that already existed in partial form elsewhere.

Three operational gaps required a dedicated corporate platform.

**Catalog fragmentation.** Product specifications, images, and category placement lacked a single authoritative store. Updates in one channel did not propagate. Sales calls repeated technical details that should have been self-service.

**Multilingual inconsistency.** Uzbek, Russian, and English content diverged when maintained separately. Missing translations blocked regional buyers. Inconsistent terminology increased support load during specification review.

**No operational admin path.** Content changes required developer intervention. Business staff could not safely update product records, publish news, or adjust category structure without deployment cycles.

Poj Pro Corporate Platform centralizes catalog and content in PostgreSQL, exposes them through a React client, and provides authenticated admin flows for authorized staff. Related systems (`poj-pro-api-contracts`, `poj-pro-telegram-bots`) extend specific automation domains; the platform remains the primary public catalog and content surface.

## System Blueprint

Architecture follows a three-tier production pattern: React client, ASP.NET Core API, PostgreSQL database, deployed behind Nginx on Linux infrastructure. Business logic resides in the API layer; the database holds normalized catalog and content entities.

Request flow is vertical: Client → API → Database → Infrastructure. No separate microservice cluster. Adjacent APIs (`Poj Pro Contracts API`) integrate at the HTTP boundary when contract-specific automation is required.

**Client layer.** React renders catalog grids, product detail views, category filters, and corporate content pages. Client routing handles locale selection. Server responses drive product availability and specification display; pricing-sensitive rules remain API-side.

**API layer.** ASP.NET Core hosts REST controllers grouped by domain: products, categories, content, authentication. Role-based access separates public read endpoints from authenticated admin mutations. Validation rejects incomplete localized records before persistence.

**Database layer.** PostgreSQL stores hierarchical categories, product attributes, multilingual text columns, media references, and news articles. Indexes support search and category traversal paths measured in production. Migrations apply through controlled release process.

**Infrastructure layer.** Nginx terminates TLS and proxies to the API. React assets deploy as static build output or served bundle depending on release configuration. Production logs support post-incident review after deployments and bug fixes.

| Tooling | Git, GitHub       | Version control, release coordination                           |

**React** supports component reuse across catalog and admin surfaces. Production codebase predates TypeScript adoption on this project; typing discipline applies at API contracts and manual review.

**ASP.NET Core** provides stable middleware for authentication and structured error responses. Controllers map closely to business domains maintained by operations staff.

**PostgreSQL** holds interconnected catalog data requiring joins across categories, products, and localized fields. Relational integrity prevents orphaned products when categories are reorganized.

**Nginx** separates public traffic from application process management. Configuration changes follow the same deployment checklist as application releases.

Adjacent production systems use overlapping stack choices: `poj-pro-api-contracts` (ASP.NET Core) and `poj-pro-telegram-bots` (Python) integrate without sharing the platform database schema directly.

## Roadmap

The platform is in production. Roadmap items describe maintenance and evolution, not initial launch phases.

**Completed — Production stabilization.** Authentication hardening, critical catalog bug fixes, PostgreSQL index tuning for category and search paths, multilingual content workflow for operations staff.

**In progress — Maintainability.** Legacy module refactoring, consistent API error shapes, admin form validation aligned with public catalog rules, documentation of deployment runbook steps.

**Planned — Catalog depth.** Extended product attribute model for technical specifications common in fire safety equipment, improved filter composition on category pages, media handling standards for specification sheets.

**Planned — Integration clarity.** Documented HTTP boundaries with `poj-pro-api-contracts` and notification flows via `poj-pro-telegram-bots`. No merge into monolith unless coupling cost exceeds integration overhead.

**Deferred — Infrastructure migration.** Container orchestration evaluated; current Linux + Nginx deployment meets availability requirements. Migration triggers on measurable ops burden, not stack preference.

Version tagging follows repository release practice. Breaking API changes require coordinated client deploy to avoid public catalog regressions.

## Lessons Recorded

**Production data outlives feature code.** Catalog records accumulated business meaning over months. Migrations on live PostgreSQL required backward-compatible steps and rollback plans. Destructive schema changes were rejected when active sales cycles depended on existing product IDs.

**Multilingual content is operational work, not a one-time translation.** Uzbek, Russian, and English fields drift when updates ship in only one locale. Admin validation now treats missing required locale fields as blocking errors on publish, not warnings.

**Admin and public views must share validation rules.** Divergent rules caused public catalog to display records admins considered draft-complete. Single validation path in the API eliminated an entire class of production inconsistencies.

**Performance fixes belong to measured paths.** Category tree loading and search were optimized after identifying slow queries in production logs, not preemptive caching layers. Index additions outperformed speculative Redis introduction for current scale.

**Hotfix discipline beats feature velocity during incidents.** Production bugs affecting catalog availability were patched and deployed before refactor cleanup. Refactors scheduled separately prevented regression during sales-critical periods.

**Adjacent systems need explicit boundaries.** Contracts API and Telegram bots solve specialized automation. Keeping them separate avoided bloating the catalog monolith with message-handling and contract PDF logic. Integration failures isolate to the calling boundary.

**Legacy code improvement is incremental.** Large rewrites were rejected in favor of module-by-module extraction with production verification after each step. Maintainability gains compound without a risky stop-the-line rewrite.
