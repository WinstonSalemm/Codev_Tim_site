## 1. Brand DNA

### 1.1 Essence

Codev_Tim is the intersection of **engineering precision** and **product maturity**. It is not a personal brand in the influencer sense — it is a **system representation** of a software engineer who builds production systems.

### 1.2 Brand Equation

```
Codev_Tim = Engineering Discipline + Product Thinking + Operational Reliability
```

### 1.3 Name Architecture

```
Codev_Tim
├── Codev     → Code + Dev — engineering identity
└── _Tim      → underscore = system namespace, terminal convention, API prefix
```

**Wordmark:** `Codev` (primary) + `_Tim` (secondary)  
**Monogram:** `CT` in monospace frame, or `C_` as system prefix  
**Version badge:** always adjacent to identity — e.g., `v0.9.4`

### 1.4 Brand Metaphors (Canonical)

| Metaphor | Usage |
|----------|-------|
| **Operating System** | Overall product frame |
| **Admin Dashboard** | Visual and layout model |
| **Command Center** | Dashboard / Operations Center |
| **Control Panel** | Module navigation experience |
| **Engineering Workspace** | Visitor entry experience |

### 1.5 Brand Anti-Metaphors (Never Use)

Portfolio, resume site, personal blog, showcase, gallery, landing page (as self-description), freelancer page.

## 7. Visual Philosophy

### 7.1 Core Principle

> Nothing decorative. Every line, border, label, and icon exists because the system needs it.

### 7.2 Visual References (Inspiration Only — Do Not Copy)

Vercel, Linear, Stripe, Raycast, Apple

### 7.3 Visual Principles

| Principle | Rule |
|-----------|------|
| **Dark-first** | Base `#07090F`, layered surfaces |
| **Signal, not flood** | Accent `#F0B429` at ≤4% screen area |
| **Border depth** | 1px hairline borders, brighten on hover — not shadows |
| **Whitespace** | ~60% content / 40% void on primary views |
| **Typography-led** | Hierarchy through type, not color blocks |
| **Grid discipline** | 4px base unit, 12-column grid |
| **No glow** | No neon, no cyberpunk, no flashy gradients |
| **Texture restraint** | Optional 2% noise overlay, 4% grid lines — barely visible |

### 7.4 Typography Philosophy

- UI/Headings: Geist Sans or Inter — technical, neutral
- Data/Terminal/Code: Geist Mono or JetBrains Mono — engineering atmosphere
- Hierarchy through size + weight + color — not bold everywhere
- Numbers: tabular-nums

### 7.5 Motion Philosophy

**Principle:** Motion with purpose — communicates hierarchy and context, never decorates.

**Allowed:** fade, blur, small translate, number counter, smooth page transitions, border highlight, soft hover  
**Forbidden:** parallax, floating objects, particles, rotation, heavy glow, bounce, scroll-jacking

Durations: 120–200ms micro, 400ms max module mount. Single easing family.

## 8. Engineering Philosophy

### 8.1 Core Beliefs

- Production systems require architectural thinking before implementation
- Trade-offs must be documented, not hidden
- Technology choices serve business constraints — not trends
- Reliability is a feature, not an afterthought
- Documentation is part of engineering — not a separate activity
- Versioning and status communicate product maturity

### 8.2 Architecture Presentation Standard

Every serious product includes a **System Blueprint** with vertical flow:

```
Client
  ↓
Gateway (when applicable)
  ↓
API
  ↓
Services
  ↓
Database
  ↓
Infrastructure
```

Diagrams: SVG inline, interactive node highlighting, premium styling — not default Mermaid theme.

### 8.3 Technology Presentation Standard

- Grouped by layer: Frontend, Backend, Infrastructure, Cloud, AI, Tools
- Text labels — no logo soup
- Optional hover tooltip with usage context

## 10. Communication Principles

### 10.1 The Three Tests

1. **Product test:** Does this sound like a software product — not marketing?
2. **ELS test:** Does this follow `02_ENGINEERING_LANGUAGE.md`?
3. **Peer product test:** Could this exist in Linear, Stripe, or Vercel?

### 10.2 Communication Hierarchy

```
System voice     → UI, status, errors, notifications (default)
Engineer voice   → Engineering records, knowledge base (factual first person allowed)
Shell voice      → Terminal command/response protocol
Record voice     → Engineer profile (data-oriented, third person or neutral)
```

### 10.3 Forbidden Communication Patterns

- Passionate, guru, rockstar, ninja, expert, best, amazing, awesome
- Creative, innovative, world-class, cutting-edge
- Hire me, portfolio, my journey, my story
- Oops, uh oh, something went wrong
- Thank you so much, let's connect, get in touch

Full forbidden list: `02_ENGINEERING_LANGUAGE.md`

## 12. Things We Always Do

- Use Engineering Language System for all copy
- Maintain persistent application shell during navigation
- Show system status and version visibly
- Present architecture as System Blueprint diagrams
- Group technology by layer
- Write module headers with MODULE label + identity name
- Provide keyboard navigation and WCAG AA contrast
- Make terminal functional and professional
- Persist operator preferences (terminal state, locale, recent actions)
- Use calm, precise, system-oriented error messages
- Answer one question per module
- Version the product (semver)
- Reference canonical docs before inventing content
- Interview Timur when information is missing

---

## 13. Things We Never Do

- Present as freelancer portfolio or hire-me page
- Use logo grids, skill bars, or Dribbble-style showcases
- Use cyberpunk, neon, excessive glow, or flashy gradients
- Use emotional, sales, or superlative language
- Use decorative animations (parallax, particles, bounce)
- Use placeholder or invented biographical content
- Break persistent shell illusion with full page reload feel
- Use «Oops» or casual error language
- Add exclamation marks to UI copy
- Copy Vercel, Linear, Stripe, Raycast, or Apple layouts directly
- Invent facts about Timur not in `03_ABOUT_TIMUR.md`
- Optimize for speed over completeness in documentation

## 14. Design Principles

1. **Intentional** — every element has purpose (Admin Panel Test)
2. **Engineered spacing** — 4px grid, no arbitrary margins
3. **Layered depth** — surface system, not shadows
4. **Signal amber restraint** — accent as indicator, not decoration
5. **Typography hierarchy** — type does the work
6. **Module consistency** — same design language across all modules
7. **Progressive disclosure** — depth available, not forced
8. **Accessible by default** — focus rings, contrast, reduced motion
9. **Performance as design** — speed is premium feel
10. **Content-first** — information over ornament

---

## 15. UX Principles

1. **OS, not website** — persistent shell, module switching
2. **Contextual transitions** — direction communicates depth (peer vs drill-down)
3. **Hover explains hierarchy** — preview importance, not decoration
4. **Loading as initializing** — boot sequence, not spinners
5. **Subtle confidence** — short, predictable reactions, no bounce
6. **Instant navigation** — prefetch, ≤240ms module ready
7. **Functional delight** — pulse, memory, metrics reveal — never distracting
8. **One question per module** — clear purpose per screen
9. **System remembers** — layout, history, recent actions, origin path
10. **Scroll stays native** — no scroll-jacking, no parallax

Full experience specification principles are embedded in `00_PROJECT_VISION.md` Section 2.4 and Experience Design conversations.

## 16. Writing Principles

1. **Precision** — one term per concept, no synonym drift
2. **Minimalism** — delete everything non-essential
3. **System-first** — UI speaks as system, not as personal brand
4. **No emotion** — neutral register, no exclamation
5. **No sales** — no CTAs that sound desperate
6. **Structure** — `[Subject] + [State/Action] + [Object]`
7. **Length limits** — buttons ≤3 words, toasts 1 sentence
8. **Peer product test** — Linear / Stripe / Vercel standard
9. **Facts over claims** — metrics, not superlatives
10. **Immutable vocabulary** — ELS terms are registered, not improvised

Full specification: `02_ENGINEERING_LANGUAGE.md`

## 17. Brand Checklist

Before publishing any module, component, copy, or interaction:

### Identity
- [ ] Uses Codev_Tim naming and versioning
- [ ] Module has correct identity name (e.g., Product Registry, not Projects page)
- [ ] Does not describe itself as portfolio, blog, or landing page

### Visual
- [ ] Background `#07090F`, accent `#F0B429` within ≤4% area
- [ ] No glow, neon, cyberpunk, flashy gradients
- [ ] Borders over shadows
- [ ] Adequate whitespace
- [ ] Typography hierarchy clear

### Language
- [ ] All copy passes ELS
- [ ] No forbidden words
- [ ] No exclamation marks
- [ ] System voice for UI elements
- [ ] Passes Linear/Stripe/Vercel test

### Experience
- [ ] Persistent shell maintained
- [ ] Module answers one question
- [ ] Transitions communicate context
- [ ] Loading feels like initializing
- [ ] Hover explains hierarchy
- [ ] Keyboard accessible, WCAG AA

### Content
- [ ] All facts confirmed in canonical docs or by Timur
- [ ] No invented biography, timeline, or contact data
- [ ] Architecture diagram included for engineering records
- [ ] Technology grouped by layer

### Engineering
- [ ] Performance target >95 Lighthouse considered
- [ ] Semantic HTML
- [ ] Reduced motion respected
- [ ] i18n keys for EN/RU/UZ where applicable

### Final
- [ ] Would a CTO trust this?
- [ ] Would a developer open the terminal?
- [ ] Does it feel like an engineering OS — not a website?
- [ ] Will the visitor remember the feeling — not the colors?
