## 10. Engineering Protocols (Personal Principles)

| # | Principle |
|---|-----------|
| 01 | **Architecture before implementation** — design system structure before writing code |
| 02 | **Business value before technology hype** — technology serves product requirements |
| 03 | **Automation over repetitive work** — automate what repeats |
| 04 | **Maintainability over shortcuts** — long-term support over quick fixes |
| 05 | **Clean code over clever code** — readable systems over clever tricks |
| 06 | **Systems thinking over isolated features** — features exist within system context |
| 07 | **Documentation is part of the product** — docs ship with code |
| 08 | **Scalability from the beginning** — consider scale at design time, not after failure |

## 11. Professional Philosophy

Считаю, что хороший разработчик должен понимать не только код, но и бизнес-задачу, которую этот код решает. Предпочитаю строить системы, которые остаются поддерживаемыми и масштабируемыми в долгосрочной перспективе. Документация, архитектура и качество кода для меня являются такой же частью продукта, как и функциональность.

---

## 12. Product Philosophy

Продукт должен решать реальные задачи пользователей, а не демонстрировать технологии ради технологий. Хороший продукт развивается постепенно, сохраняя простоту использования и возможность масштабирования. Каждая новая функция должна приносить измеримую ценность.

---

## 13. Decision Making

При выборе технических решений отдаю приоритет простоте сопровождения, масштабируемости и долгосрочной поддержке проекта. Предпочитаю проверенные технологии вместо следования каждому новому тренду. Любое архитектурное решение должно быть оправдано требованиями продукта и бизнеса.

**Preferred architecture:** Modular Monolith as default. Microservices only when objectively necessary at corresponding system scale.
